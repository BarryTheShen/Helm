## 2. Database Schema

Use SQLAlchemy 2.0 async ORM. All models inherit from a declarative base.

**CRITICAL:** Use the ORM for ALL database operations. Never write raw SQL. This ensures SQLite → PostgreSQL is a one-line config change.

### Users Table

```
users
├── id: UUID (primary key)
├── username: String (unique)
├── password_hash: String
├── role: Enum ('admin', 'user') — default 'admin' for MVP
├── created_at: DateTime
└── updated_at: DateTime
```

For MVP, single-user is fine (the server operator is the admin). But architect for multi-user from the start.

### Devices Table

```
devices
├── id: UUID (primary key)
├── user_id: UUID (foreign key → users)
├── device_name: String (e.g., "Barry's iPhone")
├── device_id: String (unique, generated by iOS app)
├── config_json: JSON — per-device UI config
│   ├── tab_bar_modules: List[String]
│   ├── default_module: String
│   ├── nav_mode: 'tabs' | 'sidebar'
│   └── ... (extensible)
├── last_seen: DateTime
├── created_at: DateTime
└── updated_at: DateTime
```

### Sessions Table

```
sessions
├── id: UUID (primary key)
├── user_id: UUID (foreign key → users)
├── device_id: UUID (foreign key → devices)
├── token: String (unique, JWT or opaque token)
├── expires_at: DateTime
├── created_at: DateTime
└── is_active: Boolean
```

### Chat Messages Table

```
chat_messages
├── id: UUID (primary key)
├── user_id: UUID (foreign key → users)
├── role: Enum ('user', 'assistant', 'system')
├── content: Text
├── metadata_json: JSON (optional — embedded components, tool calls, etc.)
├── created_at: DateTime
└── device_id: UUID (foreign key → devices, nullable — which device sent it)
```

### Calendar Events Table

```
calendar_events
├── id: UUID (primary key)
├── user_id: UUID (foreign key → users)
├── title: String
├── description: Text (nullable)
├── start_time: DateTime
├── end_time: DateTime
├── color: String (nullable)
├── location: String (nullable)
├── is_alert: Boolean (default false)
├── alert_message: String (nullable)

### Notifications Table

```
notifications
├── id: UUID (primary key)
├── user_id: UUID (foreign key → users)
├── severity: Enum ('info', 'warning', 'error', 'success')
├── title: String
├── message: Text
├── actions_json: JSON (nullable — list of action buttons)
├── is_read: Boolean (default false)
├── is_dismissed: Boolean (default false)
├── created_at: DateTime

```

### Agent Config Table

```
agent_config
├── id: UUID (primary key)
├── user_id: UUID (foreign key → users)
├── agent_api_url: String (e.g., https://api.openai.com/v1)
├── agent_api_key: String (encrypted)
├── agent_model: String (nullable, e.g., gpt-4o)
├── mcp_enabled: Boolean (default true)
├── created_at: DateTime

```

### Module State Table

```
module_state
├── id: UUID (primary key)

## 3. API Endpoints

All endpoints return JSON. Use Pydantic schemas for request/response validation.

### Authentication

**`POST /auth/setup`** — First-time server setup

- Body: `{ "username": "admin", "password": "..." }`
- Creates the first admin user
- Returns: `{ "user_id": "...", "message": "Setup complete" }`
- Only works once (returns 409 if already set up)

**`POST /auth/login`** — Authenticate

- Body: `{ "username": "...", "password": "...", "device_id": "...", "device_name": "..." }`
- Returns: `{ "session_token": "...", "expires_at": "...", "user_id": "..." }`
- Creates/updates device record
- Issues JWT session token

**`POST /auth/refresh`** — Refresh session token

- Header: `Authorization: Bearer {token}`
- Returns: new session token

**`POST /auth/logout`** — Invalidate session

- Header: `Authorization: Bearer {token}`
- Marks session as inactive

**`GET /auth/status`** — Check if server is set up and auth status

- No auth required
- Returns: `{ "setup_complete": true/false, "server_name": "...", "version": "..." }`

### Modules

**`GET /api/modules`** — List all available modules

- Returns: `{ "modules": [{ "id": "chat", "name": "Chat", "icon": "💬", "enabled": true }, ...] }`

**`GET /api/modules/{moduleId}/state`** — Get current SDUI state for a module

- Returns the full SDUI JSON payload for that module
- Used by pull-to-refresh on non-chat modules
- Returns: `{ "type": "calendar", "props": { ... }, "version": 5 }`

**`POST /api/modules/{moduleId}/action`** — Submit a user action from a module

- Body: varies per module (e.g., form submission, calendar event creation)
- Returns: updated module state

### Device Config

**`GET /api/devices/config`** — Get device-specific config

- Returns: `{ "tab_bar_modules": [...], "default_module": "chat", "nav_mode": "tabs" }`

**`PUT /api/devices/config`** — Update device config

- Body: `{ "tab_bar_modules": [...], "default_module": "...", "nav_mode": "..." }`

### Calendar

**`GET /api/calendar/events`** — Get events (with date range filter)

- Query params: `start_date`, `end_date`
- Returns: list of events

**`POST /api/calendar/events`** — Create event

- Body: event data
- Returns: created event

**`PUT /api/calendar/events/{eventId}`** — Update event

**`DELETE /api/calendar/events/{eventId}`** — Delete event

### Notifications

**`GET /api/notifications`** — Get all notifications

- Query params: `unread_only` (boolean)
- Returns: list of notification objects

**`PUT /api/notifications/{id}/read`** — Mark as read

**`PUT /api/notifications/{id}/dismiss`** — Dismiss

### Agent Settings

**`GET /api/agent/config`** — Get agent configuration

**`PUT /api/agent/config`** — Update agent configuration

- Body: `{ "agent_api_url": "...", "agent_api_key": "...", "agent_model": "..." }`

**`POST /api/agent/test`** — Test agent connection

- Sends a test message to the configured agent API
- Returns: `{ "success": true, "response": "..." }` or error

### Forms

**`POST /api/forms/submit`** — Submit form data

- Body: `{ "form_id": "...", "data": { field_name: value, ... } }`
- Backend validates and processes
- Returns: result + optional updated module state

## 4. WebSocket Server

### Endpoint

`wss://{server}/ws?token={sessionToken}&device={deviceId}`

### Connection Flow

1. Client connects with token + device ID in query params
2. Server validates token → if invalid, close with 4001
3. Server registers connection in `WebSocketManager`
4. Server sends initial state: `{ "type": "connected", "modules": [...] }`
5. Bidirectional messaging begins

### WebSocket Manager

Manages all active WebSocket connections. Key responsibilities:

- Track connections per user per device
- Route messages to the correct connection
- Handle disconnect + cleanup
- Broadcast to all devices of a user (for multi-device sync)

### Message Format (server → client)

**Chat response token (streaming):**

```json
{ "type": "chat_token", "content": "Here's", "messageId": "msg_003" }
```

**Chat response complete:**

```json
{
  "type": "chat_complete",
  "messageId": "msg_003",
  "content": "Here's your schedule for today:",
  "embeddedComponents": [ ... ]
}
```

**Chat error:**

```json
{ "type": "chat_error", "error": "Could not reach AI agent", "code": "AGENT_UNREACHABLE" }
```

**Pong (heartbeat):**

```json
{ "type": "pong" }
```

### Message Format (client → server)

**Chat message:**

```json
{ "type": "chat_message", "content": "What's on my calendar?" }
```

**Ping (heartbeat):**

```json
{ "type": "ping" }
```

## 5. Agent Proxy Service

The backend acts as a **proxy** between the iOS app and the external AI agent.

### Chat Flow

1. User sends `chat_message` via WebSocket
2. Backend stores message in `chat_messages` table
3. Backend builds conversation context (last N messages)
4. Backend calls external AI agent via **OpenAI-compatible API**:

```python
async def forward_to_agent(messages: list, agent_config: AgentConfig):
    async with httpx.AsyncClient() as client:
        response = await client.post(
            f"{agent_config.agent_api_url}/chat/completions",
            headers={"Authorization": f"Bearer {agent_config.agent_api_key}"},
            json={
                "model": agent_config.agent_model or "gpt-4o",
                "messages": messages,
                "stream": True,  # Enable streaming
                "tools": get_mcp_tools_as_openai_format()  # Expose MCP tools
            },
            timeout=60.0
        )
        # Stream tokens back to client via WebSocket
        async for chunk in response.aiter_lines():
            # Parse SSE, extract token, send via WebSocket
            ...
```

1. Stream response tokens back to iOS app via WebSocket
2. If agent makes tool calls → execute them via MCP tools → return results to agent
3. When complete, store full response in `chat_messages` table

### Agent Tool Calls

When the AI agent wants to perform an action (e.g., create a calendar event), it makes a tool call in the OpenAI-compatible format. The backend:

1. Receives the tool call from the agent's response
2. Executes the corresponding MCP tool internally
3. Returns the tool result to the agent
4. Agent continues generating its response with the tool result

This means MCP tools serve DUAL purposes:

- Called internally by the agent proxy (during chat)
- Called externally by the agent's MCP client (for background actions)

### Error Handling

- Agent API timeout (60s) → return `chat_error` to client
- Agent API returns 4xx/5xx → return `chat_error` with human-readable message
- Agent API key invalid → return specific error code, prompt user to check settings
- Network error → retry once, then return `chat_error`

## 6. MCP Server

The backend runs an MCP (Model Context Protocol) server that exposes tools for external AI agents to call.

### MCP Server Setup

Use the official `mcp` Python SDK. The MCP server runs alongside the FastAPI server (same process or separate process — agent decides).

### MCP Transport

For MVP: **Streamable HTTP transport** (recommended by MCP SDK for remote servers).

The MCP server listens on a separate port or path (e.g., `/mcp` or port 3001).

MCP connections MUST be authenticated — the agent sends a bearer token that maps to a user's `agent_config`.

## 7. Workflow Engine

A generic workflow/automation engine. For MVP, keep it simple.

### Workflow Types

- **Type A — Simple automation:** "When X happens, do Y" (e.g., when a calendar event is created, send a notification)
- **Type B — Multi-step:** Sequential steps with conditions (e.g., check weather → if rain → send reminder to bring umbrella)
- **Type C — Scheduled:** Run on a schedule (e.g., every morning at 8am, generate daily briefing)

### Implementation

For MVP, implement a simple trigger → action system:

```python
# Trigger types
class TriggerType(str, Enum):
    EVENT_CREATED = "event_created"
    EVENT_UPDATED = "event_updated"
    FORM_SUBMITTED = "form_submitted"
    SCHEDULE = "schedule"          # cron-like
    AGENT_REQUEST = "agent_request" # AI agent triggers it

# Action types  
class ActionType(str, Enum):
    SEND_NOTIFICATION = "send_notification"
    CREATE_EVENT = "create_event"
    CALL_AGENT = "call_agent"       # Ask AI agent to do something
    UPDATE_MODULE = "update_module"
```

Workflows are stored as JSON in the database. The workflow engine checks triggers on events and schedules, and executes the corresponding actions.

The vibe coding agent should research and implement this based on the project's needs. Keep it simple for MVP — even just supporting scheduled workflows (cron) and event-triggered notifications is sufficient.

## 9. Security

- **Password storage:** bcrypt hash via `passlib`
- **Session tokens:** JWT with 30-day expiry (configurable). Stored in sessions table.
- **API key storage:** Agent API keys encrypted at rest in the database (use Fernet symmetric encryption with the `SECRET_KEY`)
- **HTTPS:** The server should be deployed behind a reverse proxy (nginx/Caddy) with TLS. The server itself runs HTTP.
- **CORS:** Configure for the mobile app's origin
- **Rate limiting:** Not needed for MVP (self-hosted, single user)
- **MCP auth:** MCP connections require a bearer token. The token is the same session token or a dedicated MCP token generated in agent settings.

## 10. Plugin / Connector Architecture (for future external services)

For MVP, no external services are connected. But architect the backend with a plugin system so they can be added later.

### Connector Interface

```python
class ServiceConnector(ABC):
    @abstractmethod
    async def authenticate(self, credentials: dict) -> bool: ...
    
    @abstractmethod
    async def fetch_data(self, query: dict) -> dict: ...
    
    @abstractmethod
    async def push_data(self, data: dict) -> bool: ...
    
    @abstractmethod
    def get_mcp_tools(self) -> list[Tool]: ...
```

Future connectors (Google Calendar, Gmail, etc.) implement this interface. Each connector registers its own MCP tools.

## 12. What NOT to Build

- ❌ External service connectors (Google Calendar, email, etc.) — post-MVP
- ❌ Multi-user support beyond basic auth architecture — MVP is single-user
- ❌ File upload/storage system
- ❌ Full-text search
- ❌ Analytics / usage tracking
- ❌ Admin dashboard / web UI
- ❌ Email sending (for password reset, etc.)
- ❌ Multiple agent connections — one agent config per user for MVP
- ❌ Multiple workspaces — one workspace per server
- ❌ Backup/restore system
- ❌ API rate limiting
