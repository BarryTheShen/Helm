# Session 7 — 2026-04-07 — Web UI Editor Rethink

## Key Decisions

1. **Puck replaced with custom-built editor** — no external DnD library. Direct manipulation interaction model (click-to-create, click-to-configure, inline popovers).
2. **3-panel editor layout** — Left (structure tree + templates) | Center (interactive canvas) | Right (contextual properties + data binding).
3. **DnD scope limited to**: row reorder, component move between cells, resize handles (row heights, cell widths). No traditional palette drag-and-drop.
4. **Dual-pane mode** — PPT-style side-by-side editing with cross-screen JSON copy/paste.
5. **AI interaction via programmatic API** — deterministic state manipulation, not physical DnD simulation. Full parallel to every visual editor action.
6. **Template system** — 5 pre-built functional templates, row-level + full-screen, partial application, import/export JSON.
7. **Editor serves both roles** — inspect/tweak AI-generated screens (primary) + create from scratch (secondary).
8. **Internal data standard confirmed** — 4-part architecture: domain-specific schemas + Notion-style extensible property system + connector translation layer + OData-inspired query layer.
9. **Data binding reinstated** (Session 6 cancellation was accidental) — per-user binding at module config time. Both editor UI and AI/MCP configurable.
10. **Data binding validation** — Component Data Contracts with 3-checkpoint system: editor-time field filtering, save-time schema validation, runtime graceful fallbacks.
11. **Multi-resolution preview** — Chrome DevTools-style device selector with presets + custom dimensions + rotate. Preview-only (no per-breakpoint layouts). True responsive breakpoints deferred to V2.
12. **ChatModule and NotesModule both bindable** to external sources. Details deferred.

### Q23 Response — Does an Existing Internal Data Standard Exist?

I researched this. Short answer: **no single universal standard does what you want.** But there are pieces you can steal from. Here's the landscape:

**Domain-specific standards that already exist:**

- **iCalendar (RFC 5545) / CalDAV** — THE standard for calendar events. Google Calendar, Apple Calendar, Outlook — they all speak this. It defines: `VEVENT` with properties like `SUMMARY` (title), `DTSTART`/`DTEND` (start/end), `LOCATION`, `DESCRIPTION`, `RRULE` (recurrence), `VALARM` (reminders), custom `X-` properties. This is mature, well-defined, and every calendar service translates to/from it.
- **vCard (RFC 6350)** — standard for contacts. Name, phone, email, address, custom fields.
- **JSON-LD / [Schema.org](http://Schema.org)** — web standard for structured data. Has schemas for `Event`, `Task`, `Note`, `Person`, etc. with defined properties. Google uses [Schema.org](http://Schema.org) to understand web content.

**Generic data access standards:**

- **OData (Open Data Protocol)** — ISO/IEC approved, OASIS standard. Originally by Microsoft. Called "ODBC for the Web." Defines a REST-based protocol for querying and manipulating data. Supports `$filter`, `$select`, `$expand`, `$orderby`, `$top`, `$skip`. Self-describing via metadata (the server tells you what properties/types exist). **This is the closest thing to a universal data query standard.** SAP, Microsoft, and many enterprise systems use it.
- **GraphQL** — Facebook's query language. Client specifies exactly which fields it wants. Very flexible for UI data binding. But it's a query language, not a data schema standard.
- **JSON:API** — spec for how REST APIs should structure JSON responses (resources, relationships, pagination, filtering). More of an API convention than a data model.

**What low-code platforms do (Retool, Appsmith, Budibase, etc.):**

Every single one has its own **proprietary** internal data model. They all say "connect your data source" and then translate it into their internal representation. None of them use a universal standard for this. They just define their own `DataSource → Schema → Rows` abstraction.

**What Notion does:**

Notion has its own internal model: `Database → Properties (typed: title, text, number, select, multi-select, date, person, relation, formula, rollup, etc.) → Pages (rows)`. Every external integration (Google Calendar sync, Notion API, etc.) translates into this model. The power is in the **property type system** — any property can be any type, and you can have as many as you want. This is exactly the "display any property, any amount" flexibility you described.

**My recommendation for Helm:**

You DO need your own internal standard. No existing standard covers your full use case. But you should **steal the best ideas:**

1. **Per-data-type schemas** that map to existing standards where they exist:
    - Calendar events → based on iCalendar/RFC 5545 field names (`summary`, `dtstart`, `dtend`, `location`, `description`, `recurrence`, `reminders`)
    - Contacts → based on vCard field names
    - Tasks → based on common task fields (title, status, priority, due_date, assignee)
    - Notes → your existing schema (title, body, author, created_at)
2. **A Notion-style extensible property system** on top of that:
    - Each data record has a `properties` map: `Record<string, PropertyValue>`
    - Each property has a type: `text | number | date | datetime | select | multi_select | person | url | checkbox | relation | file`
    - The schema (which properties exist and their types) is defined per-database/collection and stored as metadata
    - Components can display ANY property, ANY amount — because they read the schema dynamically
3. **External source connectors translate INTO this internal model:**
    - Google Calendar connector: maps Google Calendar API fields → internal event schema + properties
    - Notion connector: maps Notion database properties → internal property types (almost 1:1)
    - CalDAV connector: maps iCalendar fields → internal event schema
    - Each connector is a translation layer, not a new data model
4. **Query layer inspired by OData concepts:**
    - Components request data using filter/sort/select semantics
    - `{ dataSource: "calendar-events", filter: { date: { gte: "2026-04-01" } }, select: ["summary", "dtstart", "dtend", "location"], sort: { field: "dtstart", direction: "asc" } }`
    - This is lightweight enough for SDUI JSON, powerful enough for complex queries

This gives you: (a) industry-compatible field names where standards exist, (b) Notion-level property flexibility, (c) external sources just translate into the internal model, (d) no dependency on a heavy enterprise standard like full OData.

**How it works in the editor:**

- The canvas area renders the screen at the selected device dimensions, centered with a device frame outline
- Changing the preset instantly resizes the canvas — rows, cells, and components reflow according to their width ratios and height settings
- The **Custom** option lets you type exact pixel dimensions (e.g., your actual phone's screen resolution)
- The **Rotate** button swaps width↔height for landscape preview
- The preview dimensions are stored per-editor-session (not saved to the SDUI JSON — the JSON is resolution-independent, the preview is just for visual checking)

**What this means for the SDUI JSON and rendering:**

- The SDUI JSON itself stays **resolution-independent** — it uses percentage-based cell widths (ratios like `1:2:1`), auto/fixed row heights, and the renderer adapts to whatever screen size it's on
- The editor preview just shows you HOW that adaptation looks at different sizes
- If a layout looks bad on iPad (e.g., a single-column layout that wastes horizontal space), the user would need to either: (a) adjust the layout to work better at that width, or (b) define **breakpoint overrides** (future feature — different row configs for different size classes)
- For now, this is **preview-only** (see how it looks, adjust manually if needed). True responsive breakpoints ("show 2-column on phone, 3-column on tablet") is a V2 feature that would require per-breakpoint row definitions in the SDUI JSON schema.

### Data Binding Validation — Type Compatibility System

You're absolutely right that this needs a validation layer. Without it, someone could bind a Text component to a calendar database and get garbage. Here's how I'd design it:

**The problem:** A CalendarModule needs fields like `summary`, `dtstart`, `dtend` (event-shaped data). A Text component just needs a single string value. A Chart needs numeric aggregation data. If you bind a Calendar to a "Notes" database that has no date fields, it breaks at runtime.

**The solution — Component Data Contracts:**

Each data-bindable component declares a **data contract** — the shape of data it expects:

```tsx
// In the component registry's props_schema:
{
  type: "calendar",
  dataContract: {
    type: "calendar_events",  // contract type identifier
    requiredFields: [
      { name: "title", acceptedTypes: ["text", "title"] },
      { name: "start", acceptedTypes: ["date", "datetime"] },
      { name: "end", acceptedTypes: ["date", "datetime"] }
    ],
    optionalFields: [
      { name: "location", acceptedTypes: ["text", "url"] },
      { name: "description", acceptedTypes: ["text"] },
      { name: "color", acceptedTypes: ["select", "text"] }
    ]
  }
}

// Text component — much simpler contract:
{
  type: "text",
  dataContract: {
    type: "single_value",
    requiredFields: [
      { name: "value", acceptedTypes: ["text", "number", "title", "url", "email", "phone"] }
    ]
  }
}

// Chart component:
{
  type: "chart",
  dataContract: {
    type: "aggregation",
    requiredFields: [
      { name: "value", acceptedTypes: ["number"] }
    ],
    optionalFields: [
      { name: "label", acceptedTypes: ["text", "title", "select", "date"] },
      { name: "group", acceptedTypes: ["select", "multi_select", "text"] }
    ]
  }
}
```

**Validation flow (3 checkpoints):**

**Checkpoint 1 — Editor-time (when binding is configured):**

- User selects a data source for a CalendarModule
- Editor reads the data source's schema (list of properties + their types)
- Editor checks: does this schema have fields that match the calendar's `requiredFields`?
    - If the data source has a `date` type field → can map to `start` ✅
    - If the data source has NO date fields at all → **warning**: "This data source has no date fields. Calendar requires start and end dates."
- The field mapping UI only shows **compatible fields** in each dropdown:
    - "Title field →" dropdown only shows `text` and `title` type properties from the data source
    - "Start field →" dropdown only shows `date` and `datetime` type properties
    - Incompatible properties are hidden or grayed out
- If ALL required fields can be mapped → green checkmark ✅
- If some required fields can't be mapped → yellow warning ⚠️ with explanation
- If the data source is fundamentally incompatible (no date fields for calendar) → red error ❌

**Checkpoint 2 — Save-time (when editor saves or AI sets binding):**

- The `POST /api/sdui/validate` endpoint validates the SDUI JSON
- For each component with a data binding, it checks:
    - Does the bound data source exist?
    - Are all required field mappings present?
    - Do the mapped fields' types match the contract's `acceptedTypes`?
- Returns validation errors: `"CalendarModule in Row 1 Cell 1: field mapping 'start' points to property 'Notes' which is type 'text', but calendar requires type 'date' or 'datetime'"`

**Checkpoint 3 — Runtime (when the module renders on the device):**

- The renderer fetches data from the bound source
- If a required field returns `null` or the wrong type → graceful fallback:
    - Calendar with no start date → shows "No events" placeholder
    - Chart with no numeric data → shows "No data" placeholder
    - Text with wrong type → attempts to `.toString()` the value, falls back to empty string
- Runtime errors are logged but never crash the UI

**For your specific example — Text component bound to a calendar database:**

- Text's contract says: `requiredFields: [{ name: "value", acceptedTypes: ["text", "number", "title", ...] }]`
- When binding to a calendar database, the field mapper shows ALL compatible fields from that database (title, location, description — all text-type fields) ✅
- But it would NOT show `dtstart` in the "value" dropdown because dates aren't in `acceptedTypes` for Text's `value` field (unless we add date-to-string coercion)
- So the validation naturally prevents nonsensical bindings by filtering the field mapper dropdowns

**The key insight:** Validation happens at the **field mapping level**, not at the data source level. A Text component CAN bind to a calendar database — it just can only map to text-compatible fields within that database (like the event title or description). It can't map to the start date field. The contract system enforces this automatically.
